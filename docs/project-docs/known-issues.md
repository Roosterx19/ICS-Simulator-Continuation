## 4. Character "ice-skating" + start/stop stutter

**Reported symptom**: walk animation plays but feet slide rather than
plant. Start/stop transitions are abrupt — visible jolt when walk begins
or ends.

**Code state on inspection**:
- `components/sim/mixamo-player.tsx:233-237` modulates
  `walk.timeScale = clamp(speedNow / NATURAL_STRIDE_RATE, 0.4, 2.5)`
  inside the `useFrame` body.
- `NATURAL_STRIDE_RATE` last tuned to 0.6 in `b50b28f` (was 0.85).
  At default `WALK_SPEED = 1.4` this gives timeScale ≈ 2.33 — the
  animation should play 2.33× its natural rate.
- Crossfade exists between idle and walk: `walk.fadeOut(0.2)` and
  `idle.reset().fadeIn(0.2).play()` on arrival, similar on start.

**Hypotheses for next session**:
1. **`NATURAL_STRIDE_RATE` is still mis-tuned**. If feet over-step
   ("running in place" look) at 0.6, the animation is now too fast
   relative to translation; raise back toward 0.8. If feet under-step
   (still sliding), drop further toward 0.4.
2. **Mixamo "walking" clip has root motion baked in.** The character's
   world translation is computed independently of the clip; if the clip
   ALSO translates the root, the two compound and feet slide regardless
   of timeScale. Could check by inspecting the clip's animation tracks
   for a root position channel and stripping it if present.
3. **Stutter on start/stop**: 0.2s crossfade is short. Mixamo idle is
   typically a slight sway at the same scale as walk; mismatched
   crossfade durations between fadeOut and fadeIn can produce a
   visible jolt. Worth trying matched 0.3-0.4s on both sides.

**Last attempted fixes**:
- `c407697` initial timeScale modulation, NATURAL_STRIDE_RATE = 0.85
- `b50b28f` retuned to 0.6

**To proceed**: capture user's actual perceived symptom direction
(over-stepping vs sliding) at the current 0.6 setting, then iterate
the const. For stutter, frame-grab the start moment; if the visible
jolt is brief (1-2 frames), it's the crossfade; if longer (10+ frames),
it's something else (clip blending issue, possibly a position lerp
discontinuity).

## 6. Desktop shadows not rendering after device-detection fix

**Reported symptom**: scene appears flat-lit on desktop. No shadows
under buildings or character.

**Code state on inspection**:
- `lib/utils/is-mobile.ts` was patched in `acfcb5b` to require BOTH
  `navigator.maxTouchPoints > 0` AND a mobile UA regex match. On a
  MacBook the UA doesn't match → `isTouchDevice()` returns false →
  `!isMobile` is true → shadows enabled.
- `components/sim/scenario-scene.tsx`:
  - `<Canvas shadows={!isMobile}>` ✓
  - `<directionalLight castShadow={!isMobile}>` ✓ at line 287, with
    intensity 1.6, expanded shadow camera bounds (-3000..8000 ×
    -3000..8000), shadow-bias -0.0005
  - `<ambientLight intensity={0.5}>` and `<hemisphereLight>` for fill
- All meshes have correct `castShadow` / `receiveShadow` flags
  (city-ground receives, blocks/buildings/character cast+receive).
- `b50b28f` adds a `[device]` startup log so the actual `isTouchDevice()`
  return value surfaces in the T-overlay at scene mount.

**Hypotheses for next session**:
1. **`isTouchDevice()` is still returning true on the user's Mac for
   reasons not visible in the source.** The `[device]` startup log
   added in `b50b28f` will tell us definitively — `isTouchDevice=true`
   in the overlay confirms the heuristic is still wrong. Then the fix
   is to require UA match WITHOUT touchPoints fallback for desktop
   browsers.
2. **`isTouchDevice()` returns false (correct) but shadows still
   absent.** Then the bug is downstream — possibly:
   - Ambient + hemisphere light intensity is washing out directional
     shadows. Lower ambient to 0.2-0.3.
   - Shadow camera frustum doesn't cover where the character is
     standing (would need to verify the actual character position
     falls inside [-3000..8000]).
3. **Shadow-bias too negative** at -0.0005 causes peter-panning
   (shadows detached from objects). Try 0 or -0.0001.

**Last attempted fixes**:
- `0b4500a` initial DirectionalLight + PlayerBlobShadow setup
- `acfcb5b` device-detection fix requiring UA match
- `e97d725` blob shadow Y bumped 0.06 → 0.25 to clear block mat
- `b50b28f` added `[device]` startup diagnostic log

**To proceed**: hard-reload the scenario, press T, look for the
`[device] isTouchDevice=` line at the top of the log. If it shows
true → device fix needs another pass. If false → walk through hypothesis
2/3 above.

## 7. T-overlay diagnostic color coding gap

**Reported symptom (to be specified)**: classify() in the T-overlay
isn't color-coding some log lines as expected. **Specific evidence
needed before this can be debugged**: paste of a log line + the color
it actually rendered + the color the user expected.

**Code state on inspection**:
- `components/dev/diagnostic-overlay.tsx`'s `classify()` function
  detects in this order:
  1. Explicit tag prefixes `[ERROR]/[FAIL]` → fail (red)
  2. `[OK]/[PASS]/[SUCCESS]` → pass (green)
  3. `[WARN]` → warn (yellow)
  4. Word-level `FAIL`/`fail`/`✗` → fail
  5. Word-level `PASS`/`pass`/`✓` → pass
  6. Word-level `OK` → pass
  7. Word-level `WARN`/`warn`/`warning` → warn
  8. Otherwise: returns the base level from console method (info/warn/error)
- Tag prefixes are anchored at start of string (`^\[...\]`); word-level
  matches use `\b...\b` boundaries.

**Likely failure modes worth checking once evidence arrives**:
- A log line containing `pass` as a substring of a longer word
  (e.g. `bypass`, `passport`) would mis-classify as PASS due to the
  `\b` boundary not catching internal-word matches when the surrounding
  characters aren't word-chars.
- Logs from third-party libraries that don't use the project's
  `[Tag]` convention will fall through to base-level coloring,
  which may look "uncoded" to a user expecting tag-based colors.
- The `info` base level (default `console.log`) renders as light gray
  with no row tint — easy to mistake for "uncoded" when the design
  is "info means no special treatment."

**Last attempted fixes**:
- `b50b28f` added the explicit `[ERROR]/[WARN]/[OK]/[PASS]` tag-prefix
  detection (Phase C3 Bug 5 spec).

**To proceed**: collect a specific log line that's mis-coloring,
along with what color the user expects vs. what they see. Without
that, this is unreproducible.
